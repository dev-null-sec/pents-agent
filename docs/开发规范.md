# 开发规范

## 目标

让项目在 AI 协作开发时保持清晰、可验证、可追溯。不要靠聊天记录当唯一上下文，关键状态必须沉淀到文档里。

## 编码前

每次开发前必须先给出简短计划：

```md
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

计划至少说明：

- 当前假设
- 成功标准
- 修改范围
- 验证方式
- 不做什么

## 编码中

- 优先沿用项目现有结构和风格。
- 不添加未被要求的功能。
- 不为只用一次的代码做抽象。
- 不重构无关代码。
- 发现不相关问题可以记录或提醒，但不要顺手改。
- 如果你的修改导致 import、变量、函数变成无用代码，应一并移除。

## 验证

任务完成前至少做一种验证：

- 跑相关测试
- 跑构建或类型检查
- 手动复现核心流程
- 对文档类变更做链接、目录和内容一致性检查

验证结果要写清楚：

- 运行了什么命令或检查
- 结果是什么
- 没跑什么，为什么没跑

## 文档同步

以下情况必须同步文档：

- 当前任务状态变化：更新 `docs/计划看板.md`
- 任务完成：归档到 `docs/归档计划看板.md`
- 项目当前能力变化：更新 `docs/当前状态.md`
- 重要技术或产品取舍：更新 `docs/决策记录.md`
- 长期方向变化：更新 `docs/项目路线/`
