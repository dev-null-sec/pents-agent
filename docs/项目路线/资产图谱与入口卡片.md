# 资产图谱与入口卡片

## 目标

资产搜集阶段不能只产出散乱列表。项目需要把域名、IP、URL、服务、应用、指纹、API、证据和风险线索连成一张轻量资产图谱，再把可测试入口整理成给真人和 Claude Code 都能快速理解的入口卡片。

本路线只定义 Markdown 记录模型和模板，不引入数据库，也不要求 CLI 立即实现。

## 设计原则

- 先清楚，再自动化：先用 Markdown 表达稳定结构，后续再让 `pents` 读取和合并。
- 入口优先：渗透测试真正要行动的是“可测试入口”，不是孤立的子域名或 IP。
- 证据可追溯：每个关键判断都要能回到 `evidence.md` 或 run 级证据。
- 授权状态显式化：发现资产不等于可测资产，尤其是源站 IP、第三方服务和认证后功能。
- 不把全部信息塞进一张表：顶层 `inventory.md` 负责索引，入口卡片负责细节。

## 核心实体

| 实体 | ID 前缀 | 含义 | 主要字段 |
| --- | --- | --- | --- |
| Domain | `D-` | 根域或授权域 | 域名、授权状态、来源 |
| Subdomain | `S-` | 子域名或主机名 | FQDN、DNS 状态、来源、归属 Domain |
| IP | `IP-` | 解析或线索 IP | IP、ASN/云厂商/CDN、授权状态 |
| URL | `U-` | 可访问 URL | URL、方法、状态码、需认证、入口类型 |
| Service | `SV-` | IP/端口上的服务 | IP、端口、协议、服务指纹、授权状态 |
| App | `APP-` | 一个 Web/API 应用 | 名称、框架、版本、部署入口、登录注册 |
| Fingerprint | `FP-` | 技术、框架、WAF/CDN、开源应用指纹 | 类型、值、置信度、证据 |
| API | `API-` | API 端点或接口组 | 方法、路径、参数、认证、来源 |
| Evidence | `E-` | 证据 | 已由 `evidence.md` 管理 |
| Risk | `RISK-` | 候选风险或测试建议 | 测试面、价值、前置条件、关联 finding |

ID 不要求一次性全自动生成。当前阶段可以在入口卡片中使用稳定人工编号，后续由 `pents` 生成。

## 关系模型

| 关系 | 示例 | 用途 |
| --- | --- | --- |
| `domain -> subdomain` | `D-0001 -> S-0001` | 根域和子域归属 |
| `subdomain -> ip` | `S-0001 -> IP-0001` | DNS 解析和 CDN 判断 |
| `subdomain -> url` | `S-0001 -> U-0001` | HTTP/HTTPS 入口 |
| `url -> app` | `U-0001 -> APP-0001` | 多入口可能属于同一应用 |
| `url -> fingerprint` | `U-0001 -> FP-0001` | 技术栈、WAF、框架 |
| `app -> api` | `APP-0001 -> API-0001` | 应用暴露接口 |
| `service -> fingerprint` | `SV-0001 -> FP-0002` | 端口服务识别 |
| `risk -> evidence` | `RISK-0001 -> E-0004` | 风险判断来源 |
| `risk -> finding` | `RISK-0001 -> F-0001` | 候选风险升级为 finding |

关系不需要单独建图数据库。入口卡片用“关联资产”和“证据”字段表达即可。

## 入口卡片

入口卡片用于回答一个问题：这个入口是什么、能不能测、值得测什么、证据在哪里。

建议每个可测试入口一张卡片。一个入口可以是：

- 一个 Web 站点，如 `https://ai.devnu11.cn/`
- 一个 API base，如 `https://api.example.com/api/v1`
- 一个应用聚合入口，如 `APP-0001 Sub2API 管理后台`
- 一个授权服务，如 `IP-0001:8443`

入口卡片最小字段：

| 字段 | 说明 |
| --- | --- |
| 入口 ID | `AC-0001` 形式，AC = Asset Card |
| 标题 | 人能看懂的入口名称 |
| 授权状态 | `可测`、`待确认`、`禁止` |
| 入口 URL / 目标 | URL、FQDN、IP:port |
| 归属资产 | 关联 Domain/Subdomain/IP/App |
| 存活状态 | alive、redirect、blocked、unknown、dead |
| 技术栈 / 指纹 | 框架、版本、CDN/WAF、开源应用 |
| 登录 / 注册 | 有、无、未知、需账号 |
| API / 路径 | 已知 API、隐藏路径、JS 来源 |
| 交互点 | 表单、上传、搜索、支付、OAuth、管理员功能 |
| 高价值测试点 | setup、authz、IDOR/BFLA、默认配置等 |
| 风险线索 | 候选风险，不等同于确认漏洞 |
| 证据 | `E-xxxx`、截图、raw 文件 |
| 停止条件 | 针对该入口的特殊停止条件 |
| 下一步 | 交给哪个 agent / skill / run |

## 授权状态

| 状态 | 含义 | 处理 |
| --- | --- | --- |
| `可测` | 用户已授权，且目标归属清楚 | 可进入对应低频或手工测试 |
| `待确认` | 发现了线索，但边界、账号或目标归属不清楚 | 不执行主动验证，先问用户 |
| `禁止` | 明确范围外、高风险或第三方 | 只保留为排除证据 |
| `只读记录` | 可以记录公开信息，但不主动请求 | 常用于第三方 CDN/云厂商线索 |

发现资产时默认不是 `可测`。只有 scope 或用户补充授权能把它提升为 `可测`。

## 项目文档分工

| 文件 | 职责 |
| --- | --- |
| `scope.md` | 授权范围、禁止动作、速率和边界 |
| `inventory.md` | 资产索引、入口卡片索引、API/URL/账号/测试面摘要 |
| `evidence.md` | 证据编号、来源、文件 hash、采集时间 |
| `progress.md` | 本轮执行、跳过、阻塞和待确认 |
| `review.md` | 复盘、漏掉的资产、需要新增的字典/skill/tool |
| `runs/Rxxx/` | 单轮复测的临时结果、raw、报告增量 |
| `templates/asset-card.md` | 单个入口卡片模板 |

顶层文档记录累计事实；run 级文档记录某一轮执行。入口卡片可以放在 `projects/<name>/asset-cards/AC-xxxx.md`，顶层 `inventory.md` 只放索引。

## 回填规则

Agent 输出资产时，必须区分：

- 新发现资产：先进入 `inventory.md` 或 run 级 outputs。
- 可测试入口：创建或更新入口卡片。
- 候选风险：写入入口卡片的“风险线索”，证据不足时不要直接建 finding。
- 已确认漏洞：满足复现、影响、证据后再创建 `findings/F-xxxx.md`。

合并时按以下优先级去重：

1. URL 规范化后相同。
2. FQDN + 端口 + 协议相同。
3. App 指纹和标题高度一致。
4. API base 相同。

冲突时不覆盖旧记录，追加“观察差异”和证据引用。

## 与 pents 的关系

当前阶段：

- `pents run new` 继续创建 run 目录。
- `pents evidence` 继续登记证据。
- `pents merge` 可以先合并结构化 agent 输出到 inventory/progress。
- 入口卡片先由 agent 按模板手工创建或更新。

后续可单独实现：

- `pents asset card new`
- `pents asset merge`
- `pents asset graph`
- `pents asset summarize`

这些命令不是 T-0052 的交付范围。

## 示例入口卡片摘要

```text
入口：ai.devnu11.cn
类型：Web/API
授权状态：可测
存活状态：待验证
技术栈：Sub2API 线索、Cloudflare 线索、Vue SPA 线索
登录注册：待验证
API：/api/v1/setup/status 等 JS 线索
高价值测试点：setup、admin API、OAuth、支付、默认配置
证据：E-0004, E-0011
下一步：browser-test-agent / low-frequency validation
```
