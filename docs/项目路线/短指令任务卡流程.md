# 短指令任务卡流程

## 目标

让用户只说目标和边界，AI 自动完成任务申领、任务卡生成和执行交接，避免每次都要求用户写长提示词。

这个机制服务于项目的真实定位：人定边界，AI 执行；`pents` 只是 AI 的确定性机械手。

## 职责边界

- Codex / 项目开发者：负责 ai-board 入池、排期、申领、验收和归档；负责生成或更新任务卡。
- Claude Code / 测试执行主体：只读取任务卡、scope、run 文档和正式 skill，按授权边界执行渗透流程并回填项目记录。
- 执行 agent 不操作 ai-board，不编辑 `.ai-board/`、`docs/计划看板.md` 或 `docs/归档计划看板.md`。

## 输入形态

用户可以只给短指令，例如：

```text
对 *.devnu11.cn 做主动 DNS，别碰 HTTP。
```

AI 需要自己补齐流程，而不是要求用户写：

- 长 prompt
- 完整命令
- 子代理输出结构
- 回填文件清单
- ai-board 生命周期命令。用户不需要写，Claude Code 也不需要写；这部分由 Codex / 项目开发者处理。

## AI 处理流程

1. Codex 读取 `docs/计划看板.md`，判断是否已有相关任务。
2. 读取项目 `scope.md`、run brief、已有 progress/review。
3. 判断信息是否足够。
4. 信息不足时，只问 1-3 个影响执行边界的问题。
5. 信息足够时，Codex 用 ai-board add/schedule/start 或申领已有任务。
6. 用 `pents brief` 生成任务卡。
7. 让执行 agent 读取任务卡和 scope；不要让执行 agent 操作 ai-board。
8. 执行 agent 完成后回填 inventory/evidence/progress/report-delta/review；Codex 再根据回填结果 complete/archive ai-board 任务。

## 必填字段

任务卡必须包含：

- 目标：要测试什么。
- 范围：授权内和授权外。
- 禁止动作：明确不能碰什么。
- 输入缺口：缺少账号、canary、resolver、速率、URL、授权窗口等。
- 验收标准：什么算做完。
- 回填文件：结果写到哪里。
- 完成与回填要求：执行 agent 更新哪些项目记录；Codex 如何验收和归档 ai-board 任务。

## CLI 入口

使用 `pents brief` 生成任务卡：

```bash
uv run --project cli pents brief <project> \
  --agent recon \
  --task-id T-xxxx \
  --request "对 *.example.com 做主动 DNS，别碰 HTTP" \
  --objective "生成主动 DNS 枚举计划并回填证据" \
  --in-scope "*.example.com DNS 子域名枚举" \
  --out-of-scope "HTTP 探测" \
  --forbid "端口扫描" \
  --target "*.example.com|主动 DNS|使用 subdomains-main.txt" \
  --input-gap "canary 子域名" \
  --reference "skill|skills/recon/subdomain-enumeration/SKILL.md|主动 DNS checklist" \
  --acceptance "包含 canary、NXDOMAIN、resolver 自检" \
  --writeback "runs/Rxxx/outputs/active-dns-plan.md" \
  --completion "执行 agent 回填 run progress/evidence/review；Codex 验收后归档 ai-board 任务" \
  --save
```

保存路径：

```text
projects/<project>/briefs/<task-id>-<agent>.md
```

## 反问规则

只在缺少高影响信息时反问：

- 授权范围不明确。
- 禁止动作不明确。
- 主动请求速率、授权窗口或字典规模缺失。
- 账号、入口 URL、canary、resolver 等是任务必要输入。
- 目标可能越界或涉及第三方基础设施。

不要因为低影响细节阻塞任务卡生成。可以把它们写入“输入缺口”。
