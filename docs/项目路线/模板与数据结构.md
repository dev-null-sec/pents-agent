# 模板与数据结构

## 目标

模板先于 CLI 实现。`pents` 后续命令只做机械生成、读取、追加和汇总，不在命令里发明新的数据结构。

## 模板清单

| 模板 | 生成位置 | 主要用途 | 关联命令 |
| --- | --- | --- | --- |
| `scope.md` | `projects/<name>/scope.md` | 授权范围、目标、禁止动作 | `pents new`, `pents scope` |
| `progress.md` | `projects/<name>/progress.md` | 测试过程记录和经验笔记 | `pents note` |
| `inventory.md` | `projects/<name>/inventory.md` | 资产、入口卡片索引、URL、API、参数、账号、测试面 | `pents inventory`, `pents brief` |
| `asset-card.md` | `projects/<name>/asset-cards/AC-xxxx.md` | 单个可测试入口的资产图谱、授权状态、指纹、交互点、风险线索和证据 | 后续 `pents asset card` |
| `evidence.md` | `projects/<name>/evidence.md` | 证据索引、请求响应、截图、终端输出 | `pents evidence` |
| `finding.md` | `projects/<name>/findings/<id>.md` | 单个漏洞记录 | `pents finding` |
| `report.md` | `projects/<name>/report.md` | 汇总报告草稿 | `pents report` |
| `review.md` | `projects/<name>/review.md` | 项目复盘和 skill 改进建议 | `pents review` |
| `brief.md` | 临时输出或 `projects/<name>/briefs/<role>.md` | 主代理和子代理任务卡 | `pents brief` |
| `run-progress.md` | `projects/<name>/runs/Rxxx-<date-purpose>/progress.md` | 单轮测试过程、阻塞项和待归并项 | 后续 `pents run new` |
| `run-report-delta.md` | `projects/<name>/runs/Rxxx-<date-purpose>/Rxxx-report-delta.md` | 单轮报告增量，不直接覆盖总报告 | 后续 `pents report --delta` |
| `run-review.md` | `projects/<name>/runs/Rxxx-<date-purpose>/review.md` | 单轮对 skill、CLI、tools、dicts 的反馈 | 后续 `pents review --run` |

## 数据结构原则

- Markdown 是人和 AI 的主要读写界面。
- 表格字段保持稳定，方便 CLI 做轻量解析和追加。
- 正式 finding 只由主代理确认后生成。
- evidence 用 ID 串联 progress、finding 和 report。
- brief 只包含子代理需要的最小上下文。
- 顶层文档记录累计事实；`runs/` 记录单轮执行过程和增量。
- 单轮 `Rxxx-report-delta.md` 先由主代理审核，再决定是否合并到顶层 `report.md`。
- 多次复测必须使用 `R000/R001/R002...` 编号排序，run 目录和复测报告文件都带同一编号。

## 状态字段

progress/report/review 中涉及测试面状态时，优先使用固定状态：

- 已执行：本轮已经完成，并有证据或明确结论。
- 阻塞：当前不能继续，必须写明原因类型和需要用户补充什么。
- 待确认：已有线索但证据不足，不能写成 confirmed。
- 不适用：该测试面不属于当前 scope 或本轮目标。

常见原因类型：

- 授权缺失
- 工具缺失
- 目标无输入
- 网络失败
- 风险过高
- 范围外
- 等待账号
- 其他

## ID 规则

- Asset: `A-0001`
- Asset Card: `AC-0001`
- Evidence: `E-0001`
- Finding: `F-0001`
- Brief task: `B-0001`

## pents 命令映射

`pents new` 创建：

- `scope.md`
- `progress.md`
- `inventory.md`
- `evidence.md`
- `findings/`

`pents scope` 读取 `scope.md`，输出授权范围摘要。

`pents inventory` 追加或列出 `inventory.md` 中的资产、入口卡片索引、URL、API、参数、账号和测试面。

入口卡片当前由 agent 按 `asset-card.md` 模板手工创建或更新；后续再实现 `pents asset card`、`pents asset merge` 等命令。

`pents brief` 从 `scope.md`、`inventory.md` 和可选 skill/refer 路径生成任务卡。

`pents evidence` 追加 `evidence.md`，并返回 evidence ID。

`pents finding` 基于 `finding.md` 模板创建正式 finding，并引用 evidence ID。

`pents report` 汇总 `scope.md`、`inventory.md`、`findings/` 和 `evidence.md`。

`pents review` 汇总 `progress.md`、`finding` 质量、证据缺口和 skill 反馈。

## 每轮同步规则

每轮 Claude 或子代理执行结束后，主代理必须检查：

1. `inventory.md` 是否需要新增或更新资产、URL、API、参数、账号、测试面。
2. `evidence.md` 是否需要登记本轮证据摘要、文件 hash、截图或请求响应引用。
3. `findings/` 是否需要新增 finding，或改变 candidate/confirmed 状态。
4. 顶层 `report.md` 是否需要合并本轮 `Rxxx-report-delta.md`。
5. 顶层 `review.md` 是否需要合并本轮对 skill、CLI、tools、dicts 的反馈。
6. `docs/当前状态.md` 是否需要更新项目能力、阻塞项和下一步建议。

如果没有同步，必须记录原因：证据不足、等待用户确认、工具缺失、目标无输入、风险过高、重复发现或只做草稿。

## 复测分层

同一项目的第二轮及后续测试，不应继续把所有过程直接堆进顶层 `progress.md` 和 `report.md`。

推荐结构：

```text
projects/<name>/runs/R001-<date-purpose>/
  brief.md
  progress.md
  evidence.md
  outputs/
  raw/
  R001-report-delta.md
  review.md
```

详见 `docs/项目路线/复测与运行分层.md`。
