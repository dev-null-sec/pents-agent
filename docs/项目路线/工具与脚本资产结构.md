# 工具与脚本资产结构

## 目标

让项目能逐步沉淀自动化能力，但不把 `pents` CLI 变成笨重扫描器，也不让临时脚本、第三方工具和本机安装路径散落在项目根目录或用户全局目录。

## 分层

| 层级 | 目录 | 职责 |
| --- | --- | --- |
| 第三方工具资产 | `tools/third-party/` | 统一管理安装清单、源码/fork、补丁、本机二进制和缓存 |
| 自写脚本 | `tools/` | 可复用的小型自动化脚本，按 recon/evidence/dicts/common 分类 |
| CLI 编排 | `cli/` | 项目结构、记录、任务卡、合并、报告，不直接承担重扫描逻辑 |
| 项目输出 | `projects/<name>/runs/Rxxx-*/outputs/` | 单轮脚本输出、工具结果、报告增量 |

## tools 分类

```text
tools/
  third-party/
    install/
    vendor/
    patches/
    bin/
    cache/
  recon/
  evidence/
  dicts/
  common/
```

- `third-party/install/`：第三方工具 manifest、安装脚本、版本来源和本地安装说明。
- `third-party/vendor/`：项目维护或准备二开的第三方工具源码/fork 和 LICENSE。
- `third-party/patches/`：小补丁和本地修改说明。
- `third-party/bin/`：本机可执行文件目录，普通 Git 默认不提交二进制，但 CLI 优先从这里找工具。
- `third-party/cache/`：下载、编译、临时缓存，不进入版本库。
- `recon/`：JS 静态提取、DNS 去噪、HTTP 指纹摘要、API 发现结果转换。
- `evidence/`：hash、响应摘要、证据完整性检查。
- `dicts/`：候选词条提取、命中率统计、字典分层建议。
- `common/`：通用解析、路径校验、输出格式化、安全检查。

## 第三方工具资产规则

第三方工具不能再随手装到用户全局路径后只靠 PATH 找。统一规则如下：

- 源码/fork/补丁入 Git：放在 `tools/third-party/vendor/<tool>/` 或 `tools/third-party/patches/`，保留上游来源、版本、LICENSE 和本地修改说明。
- 安装清单入 Git：放在 `tools/third-party/install/`，记录 repo、构建方式、安装目标和二进制名。
- 本机运行二进制放 `tools/third-party/bin/`：用于当前机器直接运行，`pents doctor-recon` 优先检查。
- 缓存和临时包放 `tools/third-party/cache/`：不入 Git。
- 离线可迁移二进制包后续单独用 Git LFS、release artifact 或备份包承载，不混入普通源码提交。
- 全局 PATH 安装只作为 fallback，不能作为项目标准。

当前 `dnsx` 曾安装在 `C:\Users\Administrator\go\bin\dnsx.exe`。这是历史偏差，不再作为推荐路径；后续应迁移到 `tools/third-party/bin/`，如需二开则把源码/fork 放入 `tools/third-party/vendor/dnsx/`。

## 主动 DNS 工具链分工

- 主引擎：`puredns + massdns`，用于完整字典主动 DNS 枚举和 wildcard 去噪。
- 兼容替代：`shuffledns + massdns`，用于 ProjectDiscovery 风格主动 DNS 枚举。
- 复核/fallback：`dnsx`，用于 resolver 健康检查、A/AAAA/CNAME 复核、小列表解析和兜底。

`dnsx` 不再被视为完整主动 DNS 枚举的首选主引擎。`pents active-dns` 后续实现时应优先选择 `puredns + massdns`，其次 `shuffledns + massdns`，最后才是 `dnsx -l` fallback。

## 什么时候自写脚本

满足以下条件才新增脚本：

- 同类操作已重复出现，或明显会在多项目复用。
- 手工做容易漏、慢或格式不稳定。
- 输入输出清晰，能写入 run 的 `outputs/` 或顶层文档。
- 可以通过小样例测试验证。
- 默认不触发高风险请求、写操作、认证爆破、漏洞利用或越界扫描。

不满足这些条件时，先写在 skill 或 run review 里，不急着写工具。

## CLI 与 tools 的关系

`pents` 可以调用 tools，但只做薄封装：

- 读取 scope/run 路径。
- 检查输入文件存在。
- 调用工具脚本。
- 把输出登记到 evidence 或 report-delta。

脚本核心逻辑留在 `tools/`，不要塞进 `cli/src/pents/cli.py`。

## 输出约定

工具输出默认写入：

```text
projects/<name>/runs/Rxxx-<date-purpose>/outputs/
```

原始敏感或体积较大的内容写入：

```text
projects/<name>/runs/Rxxx-<date-purpose>/raw/
```

输出优先使用：

- Markdown 摘要，方便人和 AI 读。
- JSON 结构化结果，方便 CLI merge。

## 安全约束

- 第三方工具源码/fork 可以入仓；本机二进制默认只放 `tools/third-party/bin/`，不进普通 Git。
- tools 不默认读取或写入项目根目录外的路径。
- 涉及主动请求的工具必须要求 scope、授权窗口、速率或并发参数。
- tools 不保存真实凭据、完整敏感响应或个人信息；必要时只保存摘要和脱敏说明。
- 发现工具能力缺口时，先进入需求池，再实现。

## 首批建议

优先考虑这些工具：

1. `tools/recon/static-js`：从已下载 JS 中提取路由、API、参数、OAuth、支付和配置线索。
2. `tools/evidence/hash`：对 JS、响应、截图等证据生成 hash 和登记摘要。
3. `tools/dicts/subdomain-stats`：统计 `subdomains-main.txt` 主动 DNS 命中率、噪声和耗时。
