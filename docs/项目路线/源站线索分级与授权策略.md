# 源站线索分级与授权策略

## 目标

源站线索很有价值，但也是最容易越界的区域。这个策略把“记录线索”和“验证线索”拆开：AI 可以在授权范围内沉淀证据，但不能因为发现了疑似源站 IP 就默认直连、扫端口或绕 CDN。

本策略约束 Claude Code 的 recon 和浏览器测试任务，也给 `scope.md`、入口卡片和任务卡提供统一口径。

## 核心原则

- 发现不等于授权：任何疑似源站 IP、第三方 IP、云厂商资产都先按线索记录。
- 默认先被动：能从公开数据、已有证据、DNS/CNAME、证书和历史资料判断的，不主动触达目标。
- 主动验证要单独确认：源站 IP 直连、Host 绑定验证、小范围端口确认都必须有明确 IP/端口/速率/窗口。
- 不绕防护：不做 CDN/WAF 绕过、验证码绕过、风控规避或高并发探测。
- 证据分层：每条线索都要写明来源、置信度、授权状态和下一步需要谁确认。

## 三层动作分级

| 层级 | 授权状态 | 动作类型 | 是否默认允许 | 输出要求 |
| --- | --- | --- | --- | --- |
| L1 被动线索 | `只读记录` | 不直接触达疑似源站或第三方基础设施 | 是 | 记录来源、时间、置信度和证据 |
| L2 低频验证 | `待用户确认` -> `可测` | 对明确授权目标做少量确认请求 | 否，需单独确认 | 记录授权依据、速率、端口、停止条件 |
| L3 高风险手段 | `禁止` | 绕过、扩大、爆破、扫描第三方或破坏性动作 | 否，默认禁止 | 只记录为禁止项，不执行 |

## L1：默认允许的被动线索

这些动作只收集公开或已有信息，不直接访问疑似源站 IP：

| 线索 | 默认处理 | 注意 |
| --- | --- | --- |
| 历史 DNS / 被动 DNS | 可记录候选 IP、解析时间、来源 | 不因历史 IP 直接访问 |
| 证书透明日志 / SAN | 可记录域名、证书指纹、签发者、时间 | 不把证书共现当成资产归属结论 |
| CNAME / CDN / 云厂商线索 | 可记录 CDN/WAF/第三方服务 | 不继续测试第三方基础设施 |
| 已有 HTML / JS / 配置 / 报错 | 可提取 base URL、回调域、API 域、IP 字符串 | IP 字符串默认是线索，不是可测目标 |
| 历史 URL / Wayback / urlscan | 可记录历史入口和路径 | 访问历史 URL 前仍需当前 scope 允许 |
| favicon 指纹 | 只基于已采集 favicon 或公开测绘结果记录 | 主动请求 `/favicon.ico` 属于 HTTP 验证 |
| 测绘搜索 | 可查询 Shodan、Censys、FOFA、ZoomEye、Hunter 等已有索引 | 不使用这些平台发起主动扫描或验证任务 |
| 反查绑定域名 | 可基于公开索引记录同 IP 绑定域名 | 不测试第三方绑定域名 |

推荐状态：`origin-clue-passive` 或 `只读记录`。

## L2：需要单独确认的低频验证

以下动作必须在任务卡或 `scope.md` 中明确写出授权对象、速率、窗口和停止条件：

| 动作 | 最低确认项 | 允许后怎么做 |
| --- | --- | --- |
| 源站 IP 直连 HTTP/HTTPS | IP、端口、Host、请求预算、授权窗口 | 低并发、少量 GET/HEAD，记录状态码/标题/证书/响应差异 |
| Host 绑定验证 | IP、Host header、端口、协议、请求预算 | 只验证是否绑定目标 Host，不做路径 fuzz 或 payload |
| 源站 TLS 证书确认 | IP、端口、SNI、授权窗口 | 只取证书摘要和 SAN，不扩大端口 |
| 小范围端口确认 | 明确 IP 或 FQDN、端口列表、速率 | 只测任务卡列出的端口，记录 open/closed/filtered |
| CDN 前后响应差异对比 | CDN 入口、疑似源站、请求预算 | 只对同一路径做少量对比，不绕验证码或风控 |
| 主动 favicon 获取 | 目标 URL 或授权 IP、请求预算 | 只取 favicon/hash，不做目录发现 |

推荐状态：确认前为 `origin-candidate-needs-confirmation`，确认后为 `origin-authorized-low-frequency`。

## L3：默认禁止的高风险手段

除非另有专项书面授权，本项目默认禁止：

- 扫描 CDN、云厂商、IDC 或第三方网段来找源站。
- 对疑似源站做全端口扫描、Top1000 端口扫描或高速扫描。
- 大规模 Host header 枚举、Host 绑定爆破或虚拟主机爆破。
- 绕过 CDN/WAF、验证码、风控或访问控制。
- 使用漏洞 payload、目录大字典、参数 fuzz 来验证源站。
- 对第三方绑定域名、同 IP 邻居站点或测绘结果中的非授权资产做测试。
- 伪造来源、隐藏身份、规避限速、压测或 DoS。

推荐状态：`禁止` 或 `out-of-scope-third-party`。

## 记录字段

每条源站线索至少记录：

| 字段 | 说明 |
| --- | --- |
| 线索 ID | 如 `ORIGIN-0001` |
| 候选值 | IP、Host、CNAME、证书、测绘记录或历史 URL |
| 来源 | 历史 DNS、证书、favicon、测绘搜索、JS、响应头等 |
| 采集方式 | 被动 / 已有证据 / 低频验证 |
| 授权状态 | 只读记录、待确认、可测、禁止 |
| 置信度 | low / medium / high |
| 证据 | `E-xxxx`、文件 hash、截图或 raw 路径 |
| 下一步 | 需要用户确认的 IP、端口、Host、速率和窗口 |

## 给 Claude Code 的执行口径

当任务中出现源站线索时，Claude Code 应按下面顺序处理：

1. 先判断线索来源是否属于 L1，被动线索只记录，不验证。
2. 如果需要访问 IP、发 Host header、连端口或对比 CDN/源站响应，检查任务卡是否明确授权。
3. 未明确授权时，写入 `blocked_or_skipped`：`origin_validation_requires_confirmation`。
4. 已授权时，只按任务卡列出的 IP、Host、端口、请求预算和速率执行。
5. 任一目标跳出 scope、触发 WAF/验证码/429/5xx、出现敏感数据或需要扩大端口/路径，立即停止。

## 与其他流程的关系

- `subdomain-enumeration`：可以记录 CNAME、CDN、历史 DNS 和测绘搜索线索；源站 IP 直连和端口验证必须走 L2。
- `browser-test-agent`：页面跳转到第三方或疑似源站 IP 时停止并记录，不自行扩展测试。
- `asset-card`：IP/Service/源站线索必须显式写授权状态，默认不是 `可测`。
- `T-0022` 低频验证：只覆盖已发现子域名的 HTTP/CDN、服务指纹和小范围 Web 端口；不自动覆盖源站 IP 直连。

## 验收检查

任务卡中如果包含源站验证，至少应能回答：

- 哪些 IP / Host / 端口被明确授权？
- 请求速率、并发、预算和授权窗口是什么？
- 是否允许 Host 绑定验证或 IP 直连？
- 是否允许端口扫描？端口范围是什么？
- 出现 CDN/WAF/验证码/403/429/5xx 时如何停止？
- 发现第三方资产时如何记录和排除？
