## 2026-06-02 v1 收紧为 Web/API MVP

- 背景：当前目录已有方向文档、Xalgorix 参考项目和批量导入的 skill 原料，但 `templates/`、`cli/` 仍处于早期状态。继续追求全量 300+ skill 会让第一版范围过大。
- 选择：v1 聚焦 Web 漏洞、API 安全、信息收集；先做 20-30 个高质量核心 skill、项目模板和最小 `pents` CLI。
- 不选择：不把第一版做成大而全渗透平台，不全量治理 300+ skill，不复刻 Xalgorix 的 Web UI。
- 原因：项目的优势是 Claude Code 原生工作流和 Markdown 记录闭环，不是自研扫描平台。先跑通一次真实授权测试，比堆功能更重要。
- 影响：后续任务优先级调整为模板 -> 核心 skill 治理 -> 最小 CLI -> 实战验证。

## 2026-06-02 pents CLI 采用轻量生命周期设计

- 背景：用户指定参考 `D:\Backup\Documents\服务器远程运维\ai-board` 的 CLI 设计哲学。
- 选择：`pents` CLI 只做稳定状态迁移、模板生成和报告汇总，围绕 `new`、`finding`、`report`、`review` 等生命周期命令设计。
- 不选择：不让 CLI 替代 Claude Code 做智能判断，不把 CLI 做成扫描器、代理工具、exploit 执行器或大型项目管理系统。
- 原因：ai-board 的有效之处在于少做事但保留权威状态；`pents` 也应服务于“测试记录可恢复、报告可生成、经验可反馈”。
- 影响：CLI 每个命令都应可测试、可复现、输出给人读；高风险测试动作仍由授权测试者在 scope 内显式确认后执行。

## 2026-06-02 pents 增加提效层和子代理协作层

- 背景：用户提出 `pents` 不应只用于管理项目，也应辅助 AI 提效渗透；Claude Code 支持开子代理 agent，可以用于并行拆分测试面。
- 选择：`pents` 定位为上下文压缩器、证据登记器和子代理协作辅助器；在管理层之外规划 `scope`、`brief`、`inventory`、`evidence`、`merge`、`review-agent-output` 等能力。
- 不选择：不把 `pents` 做成自动扫描器，不让子代理自由扩大 scope，不允许子代理绕过主代理直接写正式 finding。
- 原因：渗透流程最容易浪费在重复读上下文、重复测试和证据丢失；CLI 更适合把这些状态压缩、登记和合并，而不是替代测试判断。
- 影响：后续 CLI 设计要先考虑主代理和子代理的任务卡、输出格式、证据引用和合并去重。
