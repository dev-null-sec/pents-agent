- 影响：后续 CLI 设计要先考虑主代理和子代理的任务卡、输出格式、证据引用和合并去重。

## 2026-06-03 第三方工具采用源码可入仓、本机二进制本地化管理

- 背景：`dnsx` 曾被安装到 `C:\Users\Administrator\go\bin\dnsx.exe`，没有进入项目统一工具目录；同时项目后续可能二开第三方 recon 工具，并希望能打包、备份和迁移到其他电脑复用。
- 选择：第三方工具按资产形态分层管理，并统一收进 `tools/third-party/`：安装清单和脚本放 `tools/third-party/install/` 并进入 Git；需要二开的源码/fork/补丁放 `tools/third-party/vendor/` 或 `tools/third-party/patches/` 并进入 Git；本机运行二进制统一放 `tools/third-party/bin/`，`pents doctor-recon` 优先检查这里，再检查 PATH；下载和构建缓存放 `tools/third-party/cache/`。
- 不选择：不继续依赖用户全局 PATH 作为标准安装位置；不把不同平台编译出的 exe 和临时缓存直接混入普通源码提交；不把 `dnsx` 继续当作完整主动 DNS 枚举主引擎。
- 原因：源码/fork 入仓才能支撑二开和可追溯；本机二进制本地化能让 AI 和 CLI 使用稳定路径；运行产物和源码分开，才能兼顾可迁移和仓库长期健康。
- 影响：后续安装 recon 工具时默认写入 `tools/third-party/bin/`；如需修改第三方工具，先把源码或 fork 放入 `tools/third-party/vendor/<tool>/`；如果必须离线携带二进制，后续单独使用 Git LFS、release artifact 或备份包方案。

## 2026-06-03 主动 DNS 工具链采用 puredns/massdns 主引擎

- 背景：devnu11 主动 DNS 前置验证暴露出 `dnsx -d -w`、resolver 健康和不同 shell 环境下吞吐不稳定等问题；完整字典主动枚举需要更专门的高性能 DNS 工具链。
- 选择：完整主动 DNS 枚举优先使用 `puredns + massdns`；ProjectDiscovery 兼容替代为 `shuffledns + massdns`；`dnsx` 保留为 canary、resolver 健康检查、A/AAAA/CNAME 复核、小列表解析和 fallback。
- 不选择：不再把 `dnsx` 作为完整字典主动 DNS 枚举的默认主引擎；不允许跳过 canary、随机 NXDOMAIN 和 resolver 自检直接跑全量字典。
- 原因：`puredns/massdns` 更适合作为主动 DNS 批量解析主链路；`dnsx` 更适合做复核和轻量解析。工具职责分清后，AI 复测时更容易解释慢、漏、误判和工具缺失。
- 影响：`pents doctor-recon` 需要把 `puredns`、`massdns` 标成主动 DNS 主链路核心；后续 `pents active-dns` 自动选择顺序应为 `puredns + massdns` -> `shuffledns + massdns` -> `dnsx -l` fallback。该选择已被 2026-06-04 的 `massdns direct` 默认链路决策取代，保留为历史基线。

## 2026-06-04 主动 DNS 默认链路改为 massdns direct

- 背景：Claude Code 复测时，Git Bash 和 PowerShell 原生执行 `puredns bruteforce -b massdns.exe` 都出现长时间卡住；进程诊断显示 `puredns.exe` 和 `massdns.exe` 基本空转，puredns 临时文件为 0 字节，massdns 未进入 UDP 查询阶段。直接用 massdns 文件输入的 canary 可以正常完成。
- 选择：`pents active-dns --engine auto` 默认生成 `massdns direct` 计划，由 `tools/recon/active-dns-massdns.ps1` 生成候选文件并把文件直接交给 massdns；脚本负责 canary、resolver 自检、NXDOMAIN/wildcard 检查、完整枚举、命中提取和耗时记录。
- 不选择：不再把 `puredns` wrapper 作为 Claude Code 默认执行器；不再让 `dnsx` 参与默认完整字典枚举；不因 `puredns/dnsx` 缺失阻塞 massdns direct 主链路。
- 原因：当前问题集中在 wrapper 到 massdns 的 stdin pipe，而不是 resolver 慢或 Bash 慢。减少 wrapper 层可以让链路更短、失败点更少，也更容易记录候选数、命中数和各阶段耗时。
- 影响：`massdns` 成为主动 DNS 核心工具；`puredns`、`dnsx`、`shuffledns` 只作为可选诊断或替代。后续 Claude Code 正式复测 devnu11 时，应读取新的 active-dns 计划，不要复用旧 puredns 命令。

## 2026-06-04 源站线索采用三层授权策略

- 背景：devnu11 流程中需要记录 CDN/WAF、历史 DNS、证书、favicon、测绘搜索和疑似源站 IP，但源站验证很容易越过用户最初授权边界。
- 选择：把源站相关动作分为 L1 被动线索、L2 低频验证、L3 高风险手段三层；历史 DNS、证书、favicon 和测绘搜索默认只读记录；Host 绑定验证、源站 IP 直连和小范围端口确认必须单独确认；CDN/WAF 绕过、全端口扫描、网段扫描和 Host 爆破默认禁止。
- 不选择：不把“发现疑似源站 IP”自动等同于“允许直连验证”；不让 Claude Code 自行扩大到第三方基础设施；不把源站验证混进普通子域名枚举或 HTTP 存活探测。
- 原因：源站线索对攻击面判断很有价值，但验证动作的风险明显高于普通被动 recon。分层后既能沉淀高价值线索，也能让任务卡明确知道什么时候必须停下来问人。
- 影响：`scope.md`、入口卡片、subdomain-enumeration skill 和 devnu11 T-0022 任务卡都应引用该策略；后续发现源站线索时默认记录为 `origin-candidate-needs-confirmation`，获得明确 IP/Host/端口/速率/窗口后才能进入低频验证。
