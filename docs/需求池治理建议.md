# 需求池治理建议

> 生成时间：2026-06-02
> 来源：devnu11-cn-e2e 第二轮低频 recon 被动执行复盘

## 本轮复盘结论

Claude Code 已完成低频 recon 的被动部分：

- 复查 5 个被动来源：crt.sh、urlscan.io、Wayback Machine、AlienVault OTX、Google 搜索。
- 5 个来源均未发现 `devnu11.cn` 子域名或 URL 记录。
- JS 静态分析识别软件为 Sub2API。
- JS 中发现 Cloudflare Turnstile 线索，结合被动来源全空，较可能存在 Cloudflare 泛解析或通配符证书场景。
- DNS/CDN 确认、端口确认、服务指纹和主动验证仍被实际 URL、授权窗口和请求速率卡住。

因此，继续追加普通被动来源的收益已经很低。下一步真正有价值的是：

1. 让用户提供实际访问 URL。
2. 在授权窗口内做低频主动入口确认。
3. 把这次实战反馈沉淀到 skill、模板和 `pents` CLI。

## 补充反馈：主动子域名枚举和工具缺失说明不足

用户进一步指出：子域名扫描仍然不给力，Claude 反馈没有清楚说明是否做了主动扫描，也没有说明缺哪些工具、是否建议下载和如何安装。

这个问题应拆成三个改进点：

1. **主动枚举闭环不足**：当前流程写了“不做大字典爆破”，但没有给出可接受的低频主动枚举方案，例如小字典、限速、解析器选择、停止条件和结果回填。
2. **工具缺失反馈不足**：如果 `subfinder`、`dnsx`、`httpx`、`shuffledns`、`subzy` 等工具缺失，Claude 必须明确说明“未执行原因=工具缺失”，并给出本机安装建议。
3. **失败反馈格式不足**：当 recon 被跳过或阻塞时，报告不能只写“未执行/无结果”，必须写清楚已尝试来源、未执行项、未执行原因、需要用户补充的信息和建议安装工具。

因此已新增需求：

| `T-0027` | 补齐主动子域名枚举闭环 | 让主动枚举有授权门槛、低频策略和反馈格式 |
| `T-0028` | 增加 recon 工具缺失检查和安装建议 | 让 Claude/pents 明确提示缺哪些工具、怎么装 |
| `T-0029` | 规范 recon 失败反馈格式 | 让 skipped/blocker 不再含糊 |

## 推荐新增需求

### P0：补齐实际入口并执行低频主动验证

原因：当前所有 finding 仍待确认，主动阶段被实际 URL、授权窗口和速率卡住。

验收建议：

- `scope.md` 记录至少 1 个实际访问 URL、授权窗口、允许请求速率。
- Claude Code 对已确认入口执行低频 DNS/CDN、端口、服务指纹。
- 验证 F-0001 的 `/setup` 和 `/api/v1/setup/status` 是否真实存在。
- 结果回填 inventory/evidence/report/review。

### P1：实战反馈驱动 recon skill 修订

原因：`subdomain-enumeration` 的被动无结果处理不足，`javascript-analysis` 在软件识别/CDN 线索方面价值被低估。

验收建议：

- `subdomain-enumeration` 增加被动无结果诊断 checklist。
- 增加 Cloudflare 泛解析、通配符证书、Turnstile 线索的判断和停止条件。
- `javascript-analysis` 增加已下载 JS 文件直接分析、软件识别和配置变量提取。
- `api-discovery` 增加 JS bundle 推断 API 结构的步骤。

### P1：为 Sub2API 类开源应用建立指纹与验证路线

原因：本次通过 JS 识别 Sub2API 后，攻击面可以从通用 Web/API 转为“开源应用指纹 + 配置/版本/默认路径验证”。

验收建议：

- 新增或扩展 recon 文档，说明如何从 JS、路由、默认配置识别开源应用。
- 记录“只阅读公开源码，不测试第三方 demo”的边界。
- 输出 Sub2API 后续验证路线：setup、admin API、OAuth、支付、默认配置。

### P1：更新模板和报告同步规则

原因：`progress/inventory/evidence/review` 已更新第二轮 recon，但 `report.md` 和 `docs/当前状态.md` 一度滞后。

验收建议：

- `templates/report.md` 增加“已执行 / blocker / 待确认”三类状态。
- `templates/progress.md` 增加 recon 阶段分层：被动完成、主动阻塞、认证后待测。
- 文档规则明确：Claude 每轮执行后必须同步 report 和 当前状态，或显式记录未同步原因。

### 已处理：清理 devnu11 项目文档小问题

原因：本次记录中有若干细节问题，例如 `Wayback Machine 无 archdevnu11.cn 归档` 疑似笔误、report 未体现第二轮 recon。

处理结果：

- 已修正 evidence 中明显笔误。
- report 已补充第二轮被动 recon 结果和 Sub2API/Cloudflare 线索。
- progress 中“需要 Claude Code 按 brief 执行低频 recon”已改成“被动部分已完成，主动部分待 URL/窗口/速率”。
